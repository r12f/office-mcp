# 09 — User Interface

`office-mcp` ships two user-facing surfaces:

- A daemon UI for the long-running local service.
- An Office add-in task pane UI for the current document runtime.

Both surfaces are product UI, not debug pages. They MUST use a modern frontend
stack with componentized state, typed view models, accessible controls, and a
maintainable design system. A static, minimally styled HTML status page is not
acceptable for production.

The visual brief is an **Office operations console**: quiet, information-dense,
and trustworthy enough to sit beside Word all day. It should feel closer to a
local command center than a landing page or developer log viewer. The signature
interaction is a live **command ledger**: every client request has a visible
life cycle from queued, to running, to completed, with the document it touched
and the outcome shown without leaking document content.

## 1. Design principles

- **Status first**: the user must understand whether the bridge is up before
  inspecting details.
- **Local diagnostics**: common support questions must be answerable from the
  UI without opening logs or a terminal.
- **No document content leakage**: history views may show tool names, high-level
  intent, target session metadata, status, duration, and error summaries; they
  MUST NOT show document body text, inserted text, extracted text, image data,
  or full tool arguments by default.
- **Bounded retention**: UI history is in-memory and bounded unless audit logging
  is explicitly enabled. The UI is a live monitor, not a permanent log store.
- **Host-aware grouping**: documents are grouped by Office app (`Word`, `Excel`,
  `PowerPoint`, `Outlook`, then future apps), with unsupported or unknown hosts
  grouped under `Other`.
- **Accessible desktop UX**: keyboard navigation, visible focus, high-contrast
  mode, screen-reader labels, reduced-motion support, and readable density are
  required.
- **Operational density over decoration**: surfaces prioritize scannable tables,
  split panes, status strips, and inspectors. Avoid marketing-style hero
  sections, oversized cards, gradient backgrounds, decorative blobs, or nested
  cards.
- **Document context without document content**: Word, Excel, PowerPoint, and
  Outlook should be visually distinguishable through restrained app accents and
  icons, but previews of document body content are prohibited.
- **Plain operational copy**: labels and errors use active voice and tell the
  user what happened and what to do next. Avoid vague errors such as
  "Something went wrong".

## 2. Visual system

The UI MUST define shared design tokens used by both the daemon window and the
add-in task pane. These tokens may adapt to light/dark themes, but the default
theme should be a calm desktop utility theme, not a monochrome or single-hue
palette.

### 2.1 Color tokens

Minimum token set:

| Token | Default value | Purpose |
|---|---:|---|
| `canvas` | `#F7F8FA` | App background |
| `surface` | `#FFFFFF` | Panels, menus, table surfaces |
| `surface-raised` | `#F2F5F8` | Selected rows and subtle grouping |
| `text-primary` | `#17202A` | Primary content |
| `text-secondary` | `#5A6673` | Metadata and secondary labels |
| `border` | `#D8DEE6` | Rules, separators, table borders |
| `word` | `#2B579A` | Word sessions and tool namespace accent |
| `excel` | `#217346` | Excel sessions and tool namespace accent |
| `powerpoint` | `#B7472A` | PowerPoint sessions and tool namespace accent |
| `outlook` | `#0078D4` | Outlook sessions and tool namespace accent |
| `success` | `#168A45` | Successful commands and Up state |
| `warning` | `#B7791F` | Degraded, stale, retrying |
| `danger` | `#C9352B` | Failed commands and Down state |
| `focus` | `#4C8DFF` | Visible keyboard focus |

App colors are accents only: left row rails, icons, badges, or tiny status
marks. They MUST NOT dominate the whole page.

Color implementation MUST pass these contrast gates in light, dark, and
high-contrast themes:

- Body text and data text: at least 4.5:1 against its background.
- Large text, icons, badge text, and app accent marks that carry meaning: at
  least 3:1 against adjacent colors.
- Focus indicators: at least 3:1 against both the component and surrounding
  background.
- Status cannot rely on color alone; use text and iconography with the color.

### 2.2 Typography

- UI font: system UI stack, starting with `Segoe UI` on Windows.
- Data font: `Cascadia Mono`, `SFMono-Regular`, `Consolas`, or equivalent for
  IDs, endpoints, durations, ports, and timestamps.
- Number columns use tabular numerals.
- Headings use compact desktop scale, not landing-page display type.
- Long IDs, file names, endpoints, and client names must truncate or wrap
  intentionally with tooltips or copy affordances where useful.

### 2.3 Motion

Motion is functional:

- Running tasks may show a subtle elapsed-time pulse or progress stripe.
- Reconnect state may show a countdown tick.
- Row expansion and panel transitions animate only opacity and transform.
- All animation respects `prefers-reduced-motion`.
- Never use `transition: all`.

### 2.4 Iconography

Use a consistent icon set such as Fluent UI icons or lucide icons. Required
icons include status, Word, Excel, PowerPoint, Outlook, client, document, task,
history, settings, copy, open, retry, warning, and close. Icon-only controls
MUST have accessible names.

## 3. Frontend architecture

### 3.1 Recommended stack

The daemon main window and add-in task pane SHOULD share a TypeScript UI package
containing design tokens, primitives, status badges, history lists, and typed
view models.

Recommended implementation:

- TypeScript + React.
- Vite or equivalent modern bundler.
- Component library owned by the repo or a small dependency set; avoid heavy
  application frameworks that complicate local packaging.
- TanStack Query, Zustand, Redux Toolkit, or an equivalent explicit state layer
  for server-derived state.
- CSS modules, vanilla-extract, Tailwind, or equivalent tokenized styling.

The exact stack is an implementation choice, but it must support production
bundling for both the desktop daemon UI and Office webview task pane.

### 3.2 UI data source

The daemon exposes a local UI state API that is separate from the public MCP
tool surface:

- The default endpoint is `https://localhost:<addin_channel.port>/ui/state` for
  snapshots and `wss://localhost:<addin_channel.port>/ui/events` for live
  updates when WebSocket transport is used.
- The API is loopback-only and MUST refuse non-loopback binds.
- Access control is limited to loopback binding and strict browser `Origin`
  validation, matching the local trust-zone model in [05-security.md](05-security.md).
- It MUST remain metadata-only. Access control is the local trust-zone model:
  loopback-only listeners plus strict `Origin` validation.
- The add-in task pane may request only the redacted daemon summary after its
  WSS registration succeeds; it MUST NOT receive other clients' identities,
  global command history, config paths, or log paths.
- Allowed origins are the configured local add-in HTTPS origin and the daemon
  window origin. Missing or foreign browser origins are rejected.
- It returns a redacted status snapshot and a bounded command history.
- It may use HTTP polling, Server-Sent Events, or WebSocket updates.
- It MUST enforce the same size limits and origin checks as other local browser
  entry points.

The UI API is not an MCP endpoint. MCP clients cannot use it for discovery,
tool calls, or document access.

The add-in task pane receives its own live state from the add-in runtime and the
daemon registration channel. It may also request a redacted daemon status
snapshot after registration succeeds.

### 3.3 Web interface requirements

The implementation MUST follow these baseline interface rules:

- Use semantic elements first: `button` for actions, links for navigation,
  `table` for tabular data, `label` for form controls.
- Every interactive element has a visible `:focus-visible` state.
- Icon-only buttons have `aria-label`; decorative icons are hidden from assistive
  technology.
- Async connection changes, command completion, and validation errors use polite
  live regions.
- Forms use real labels, meaningful `name` attributes, correct input types, and
  inline errors. The first invalid field receives focus on submit.
- Do not block paste.
- Destructive actions, including Quit, require confirmation or an undo window.
- Dates, times, durations, and numbers are formatted with `Intl.*` APIs.
- Long user-controlled strings handle short, average, and very long values
  without layout breakage.
- Lists that can exceed 50 rows use virtualization or `content-visibility`.
- The URL or desktop route reflects selected tabs, filters, and expanded panels
  when the surface supports deep links.
- Native controls render correctly in light, dark, and high-contrast themes.

### 4.1 Tray icon states

The tray icon represents daemon health:

| State | Meaning |
|---|---|
| `Up` | MCP HTTP listener and add-in HTTPS/WSS listener are bound, and the event loop is healthy. |
| `Degraded` | The daemon is running but one listener is unavailable, certificate loading failed, or required config is invalid after reload. |
| `Down` | The tray controller cannot reach the daemon process, or the daemon is stopping. |

The user-facing menu uses `Up`, `Degraded`, and `Down` consistently. When the
state is `Degraded`, the tray menu shows `Status: Degraded` and the main window
shows the failing subsystem and last error.

### 4.2 Right-click menu

Right-clicking the tray icon MUST open a native menu with this order:

1. Status line: `Status: Up`, `Status: Degraded`, or `Status: Down`.
2. Client count: `Clients: <n>`.
3. Document connection count: `Documents: <n>`.
4. Separator.
5. `Show Office MCP Control`.
6. `Quit Office MCP Control`.

The first three items are read-only. `Show Office MCP Control` opens or focuses the main
window. `Quit Office MCP Control` asks for confirmation, then tells the daemon to shut
down gracefully, closes add-in sockets with close code `4004`, and exits after
in-flight calls complete or their deadlines expire.

`Documents: <n>` is the count of connected document sessions that are currently
addressable by the daemon. Sessions marked `stale`, expired, reconnecting, or
otherwise unavailable during the grace period MUST NOT be counted as connected
documents in the tray tooltip, menu, or quit confirmation.

Closed or disconnected document sessions may appear in daemon diagnostics only
during the reconnect grace period. The effective grace period defaults to 60
seconds and MUST NOT exceed 300 seconds. Once it expires, the session is deleted
from the daemon registry and must disappear from document lists; the UI must not
retain historical session cards.

The menu MUST be a platform-native context menu, not a web-styled floating
panel or custom-drawn imitation. It must use OS-native font metrics, separators,
hover/selection behavior, disabled/read-only item behavior, keyboard access,
and light/dark/high-contrast adaptation. Any custom tray menu adapter must prove
native behavior with manual Windows evidence before release. A tray surface that
looks like an HTML menu, frameless utility window, browser popup, or toolkit demo
is a release blocker even when the click targets work.

On Windows, the tray implementation must use a native notification-area icon and
native context menu primitives. A browser window, webview, frameless HTML menu,
CSS-styled popup, or manually positioned floating panel is not acceptable even
if it visually resembles the Windows menu. The release evidence must include a
visible screenshot or recording where the menu is opened from the actual tray
icon in the notification area, with the menu visually attached to that icon.

The tray icon tooltip shows `Office MCP Control - Up - <n> clients - <n> documents` or
the equivalent Degraded / Down state. The icon state must update within two
seconds of a daemon health transition.

## 5. Daemon main window

The main window is the primary diagnostic and control surface for the daemon.
It MUST be available from the tray menu and SHOULD also be available from a CLI
command such as `office-mcp ui`.
The production daemon must serve the main window at a discoverable UI URL for
the lifetime of `daemon run`; `daemon status` and the UI-opening CLI must report
or open that exact runtime URL. Evidence fixtures may use temporary ports, but
the release gate must also prove the normal daemon path exposes `/ui/`,
`/ui/state`, and `/ui/events`.

### 5.1 Information architecture

The main window uses a durable operations-console layout:

```text
+-----------------------------------------------------------------------+
| Status Strip: Up | 2 Clients | 3 Documents | 1 Active Task | endpoints|
+----------------------+------------------------------+-----------------+
| Documents            | Activity                     | Inspector       |
| - Word               | - Current Tasks              | selected row    |
|   report.docx        | - Recent Command History     | details         |
| - Excel              |                              | recent 10       |
+----------------------+------------------------------+-----------------+
```

- The status strip is always visible.
- The document/client navigation pane filters the activity list.
- The activity pane shows current tasks above recent history.
- The inspector shows details for the selected client, document, task, or
  history entry.
- On narrow desktop widths, the inspector becomes a drawer; on small task-like
  windows, sections stack vertically in the same order.

### 5.2 Required sections

The main window MUST show:

1. **Daemon status and information**
   - Status: `Up`, `Degraded`, or `Down`.
   - Version.
   - Uptime.
   - MCP endpoint.
   - Add-in HTTPS/WSS endpoint.
   - Config file path.
   - Log file path when configured.
   - Last error when degraded or down.

The daemon summary MUST NOT expose an unexplained `Config` field. If the UI
shows configuration in the top status/details rail, it must behave like `Log`:
the value is the effective config file path as selectable text with copy/open
affordances. It may wrap or use a bounded visual abbreviation when space is
tight, but the rendered value must not become an unselectable ellipsis-only
fragment. A normal default configuration must not be shown as `Not configured`;
if the daemon is using defaults, show the actual default config file path that
would be read or written. If no config file path exists in a specific deployment
mode, omit the field from the first row and show the reason in diagnostics/
details instead of displaying `Not configured`.

2. **Connected MCP clients**
   - Client ID or session ID.
   - Transport (`http` or `stdio-bridge`).
   - User-agent or client name when available.
   - Connected time.
   - Last activity time.
   - In-flight request count.

3. **Connected documents**
   - Grouped by host app: Word, Excel, PowerPoint, Outlook, Other.
   - A compact document summary block whose primary label is the document title
     or filename. The title has first priority in the row layout and must not
     be squeezed by status badges, counts, or other right-side metadata; long
     titles use intentional middle/end truncation with a tooltip.
   - The session ID, shown as a middle-truncated compact value with a copy
     affordance or tooltip. The full session ID remains available through copy
     and the Inspector; the compact card must not render an unbounded full ID.
   - Connection state shown to the user as only `active` or `dead`, with both
     text and a scan-friendly icon or color-coded status marker. The compact
     document card must not display `reconnecting` unless the daemon has
     explicit evidence that a live reconnect attempt is currently in progress;
     stale, disconnected, unavailable, or otherwise non-addressable sessions are
     displayed as `dead`.
   - Host/add-in version from the registered Office runtime. A connected Word,
     Excel, or PowerPoint session must not show host `other` or `Version -`
     when the add-in registered host metadata successfully; if this happens,
     treat it as a daemon/add-in identity correlation bug and preserve evidence
     in logs and tests.
   - Available tool count.
   - Current queue depth.
   - Finished task count for that document session.
   - Failed task count for that document session.

Connected document blocks are navigation and status summaries, not activity
panes. Each block should fit the amount of content it contains and should not
show recent command history, endpoint details, or expanded host metadata by
default. The right-side Activity or Inspector pane owns recent commands and
detailed diagnostics. Session IDs are operationally important and must be
visible in compact form on each document card, but full IDs belong behind copy,
tooltip, or Inspector affordances.

Document connection state must be visually scannable when many sessions are
present. Use a stable icon, status dot, or host-accent-compatible color marker
next to the state label; text alone is not sufficient, and color alone is not
sufficient for accessibility. The compact document-card vocabulary is
user-facing: `active` for connected usable sessions and `dead` for disconnected,
stale, expired, unavailable, or otherwise non-addressable sessions. Do not
guess that the Office side is reconnecting and do not soften `dead` into
`reconnecting` without daemon-owned proof of an active reconnect attempt. Raw
internal states may remain inspectable in the right-side Inspector for
diagnostics.

4. **Current tasks**
   - Each in-flight command across all clients and documents.
   - Tool name.
   - Target app and session.
   - Client name or ID.
   - Start time and elapsed time.
   - Timeout/deadline.
   - High-level user intent when supplied by the client.

5. **Recent command history**
   - The most recent 10 completed commands.
   - Tool name.
   - Target app/session.
   - Client name or ID.
   - Start and completion time.
   - Duration.
   - Result: `success`, `failure`, `cancelled`, or `timeout`.
   - On failure: `office_mcp_code`, short error message, retriable flag, and
     `partial_effect` when known.

6. **Global tool access control**
   - A daemon-wide all-tools switch.
   - App-level controls for Word, Excel, and PowerPoint.
   - Category-level controls matching each host capability spec.
   - Per-tool controls for every Word, Excel, and PowerPoint tool.
   - A `Read` / `Write` / `All` access ceiling that applies before app,
     category, and per-tool toggles.
   - Clear text that daemon policy is global for all MCP clients and sessions,
     while task-pane `available_tools` remains document/session-specific.

The daemon global tool access controls must use the same object/category and
side-effect metadata as the task pane tools UI, but they apply at a different
layer. Disabling a tool in the daemon removes it from MCP `tools/list` and
causes stale calls to fail before any session lookup. Disabling a tool in a task
pane removes it only from that session's `available_tools`.

Removed sessions are not shown in Connected documents. They may appear only as
targets inside recent command history or a future disconnected-sessions view.

### 5.3 Status strip

The status strip MUST include:

- Health badge: `Up`, `Degraded`, or `Down`.
- Client count.
- Document count.
- Running task count.
- MCP endpoint with copy control.
- Add-in endpoint with copy control.

The status strip uses color and text together. Color alone is not sufficient.
The running task count label must be explicit. Do not show a bare `Running`
label, because users can confuse it with daemon health. Use `Active Tasks`,
`Tasks Running`, or `Running Tasks`, and keep the value near the label. When the
count is zero, the UI may show `0` in the same compact metric style, but the
label must still explain that it refers to task/command execution.

The top daemon details rail below or beside the status strip must allocate
space by operational importance. `Version` and `Uptime` are compact metadata
and should use fixed or content-sized columns. `Config`, `Log`, and
`Last Error` are diagnostic values and must render as selectable text with a
separate copy control. Long `Config` and `Log` paths may wrap or use a bounded
visual abbreviation, but must remain selectable and copyable. `Last Error` is
the primary diagnostic field when something is wrong and must receive the
widest flexible area; it must be able to wrap and must not be ellipsis-only or
copy-only. The layout must not give `Version`, `Uptime`, or configuration
metadata large empty columns while truncating or squeezing `Last Error`. When
there is no error, `Last Error: None` may be compact, but the grid should still
reserve the error column as the dominant diagnostic region so degraded/down
states do not reflow or hide the error message.

### 5.4 Tables and lists

Client, document, current-task, and history lists are tabular data. They SHOULD
use semantic tables or an accessible grid pattern when virtualization is needed.

Required behavior:

- Sort by status, app, client, last activity, duration, and result where useful.
- Filter by app, client, result, and text search.
- Preserve selected row when live updates arrive.
- Do not resize columns in a way that causes layout shift during polling.
- Use tabular numerals for counts, durations, queue depth, and timestamps.

When virtualization or an accessible grid pattern is used:

- Preserve keyboard row navigation with arrow keys, Home/End, Page Up/Page Down,
  Enter/Space for selection, and Escape to close drawers or inspectors.
- Preserve focus and selected row across live updates, sorting, filtering, and
  row recycling.
- Expose row count, row index, selected state, and column headers to assistive
  technology.
- Announce material async changes, such as a command completing or a session
  disconnecting, through a polite live region.
- Keep the DOM order consistent with the visual order.

### 5.5 Detail expansion

Rows for clients, documents, current tasks, and history entries MUST support a
detail expansion panel. The panel may include structured metadata and redacted
arguments, but it MUST NOT show document body content or full unredacted tool
arguments unless a future explicit diagnostics mode is added.

Document detail panels in the compact left document list MUST NOT duplicate the
right-side Activity pane. The left document block shows only summary identity
and counters: document name, state, version, available tool count, queue depth,
finished task count, and failed task count. The Activity or Inspector pane may
show that document session's most recent 10 commands and their results when the
document is selected. This is separate from the global recent command history,
which is also capped at 10.

### 5.6 Empty and failure states

- With no clients connected, show an empty clients section with the MCP endpoint
  and stdio bridge command.
- With no documents connected, show an empty documents section with the add-in
  endpoint and activation hint.
- When the daemon is degraded, keep the window usable and show the failing
  subsystem and last error.

Empty states must include a concrete next step, such as copying the MCP endpoint
or opening the add-in in Word. They must not look like blank tables or broken
loading states.

## 6. Add-in task pane UI

The add-in task pane is the document-local view. It MUST make the current
document's connection state visible without requiring the daemon main window.
It is narrow by default and must be optimized for a 320-420 px task pane.

The add-in must present as a finished product in the Office ribbon and task
pane chrome. Manifest metadata, ribbon labels, icon assets, and task pane title
must use the product identity from section 2.5. Developer-facing placeholders
such as `office-mcp`, generic `Open`, blank icons, default add-in icons, or
experimental/debug labels are not acceptable in release builds.

Required add-in presentation rules:

- `DisplayName`, ribbon group labels, and catalog labels use a stable product
  name and action-oriented wording. Inside the constrained task pane, the
  visible top-level product/category label and panel title SHOULD use the
  shorter `MCP Control` instead of repeating `Office MCP Control`; the full
  product name remains acceptable in catalog, installer, tooltip, and about
  surfaces where space permits.
- The visible add-in title, icon, and type/category must be reviewed together as
  one first-run product impression. The title cannot be a lowercase package
  slug, protocol name, host-specific implementation label, or generic task-pane
  label. The icon cannot be blank, missing, default Office/webview artwork, or a
  gear-only settings icon. The type/category cannot sound like an experiment,
  sample, MCP protocol adapter, developer tool, or internal scaffold.
- The add-in type/category shown in Office catalogs, marketplace/pre-submission
  metadata, and installer-generated manifests must describe the product as a
  local productivity automation/control utility. It must not look like an
  internal test add-in, sample, debug panel, or raw protocol bridge.
- Ribbon command icons use the generated product icon set at all Office-required
  sizes and remain legible on light/dark ribbon themes.
- Any visible `Open Control Panel` command in the add-in, ribbon, tray, daemon
  UI, or task pane uses the generated product mark as an icon. Text-only control
  panel buttons are allowed only in menu systems where the platform convention
  does not display command icons.
- The add-in description and support metadata explain a local office automation
  control surface without implying Microsoft ownership or certification.
- Host-specific catalogs for every implemented Office host render the same
  identity system while using restrained host accents only inside the task pane
  content.

Every implemented host catalog entry must show a polished product title,
product icon, concise description, provider name, host-specific support text,
and local productivity automation/control category or type metadata where the
manifest/catalog format exposes it. The catalog-visible name must be the same
product family as the ribbon command and task pane title; raw package names such
as `office-mcp-word`, lowercase implementation slugs, and generic titles such
as `Add-in`, `Open`, or `Task Pane` fail product acceptance.

The add-in first-run impression is evaluated as a single product surface. Title,
icon, provider, ribbon group, command label, task pane chrome, description, and
type/category must all be updated together. A polished title paired with a blank
icon, an original icon paired with an experimental type/category, or a mature
catalog entry paired with a generic ribbon command is still a failure. The
recommended release identity is `Office MCP Control` for the product family,
`Open Control Panel` for the primary command, and a catalog type/category that
describes a local productivity automation and control utility.

The catalog-visible type/category is part of the UI, not package metadata only.
It must be reviewed from the installed Word, Excel, and PowerPoint catalog
screens a normal user sees. Generic strings such as `Add-in`, `Task Pane`,
`Developer Tool`, `MCP Server`, `Protocol Bridge`, `Sample`, `Debug`, or raw
host package names fail acceptance because they make the add-in read as an
experimental half-finished component.

Catalog installation scripts must keep every implemented host entry in sync
with the current daemon origin and generated asset URLs, including Word, Excel,
and PowerPoint. A catalog entry that disappears after reinstall, points at stale
localhost ports, omits the icon, or shows mismatched title/type metadata fails
the release gate.

Current visible-build feedback requires another installed-surface audit for
every implemented host. Word, Excel, and PowerPoint catalog cards, ribbon
commands, task pane chrome, icons, provider text, descriptions, and visible
type/category must present one coherent product identity. Any surface that still
looks like a test add-in, generic Office container, raw MCP bridge, host scaffold,
or half-finished sideload package blocks release even when the add-in connects.

### 6.1 Information architecture

The task pane uses a compact vertical layout:

```text
+--------------------------------+
| Connection + document summary  |
|   Settings panel, when open    |
| Current Task                   |
| Recent Tasks                   |
+--------------------------------+
```

- The connection badge remains visible while scrolling.
- The first viewport must be content-dense. Header and document blocks size to
  their content and MUST NOT reserve large empty vertical space. A block may
  grow only when its content, expanded details, or current task state requires
  it.
- The connection summary and document identity are one compact top block, but
  metadata inside that block must be grouped by ownership instead of mixed into
  one long row list. Document-owned fields stay together: document title, host,
  protection, and dirty/read-only state. Daemon/session-owned fields stay
  together: daemon endpoint and reconnect control, server/protocol version,
  session ID, and last connection error. The daemon/session group should sit in
  the same visual column as the `MCP Control` title so daemon controls read as
  product controls, while document metadata reads as the current document
  context.
- Metadata rows inside the top block must use normal row height and vertical
  centering. The `Daemon` endpoint row is a compact row like `Runtime`, `Host`,
  and `Protection`; adding a reconnect icon button must not increase the row to
  card-like height. Endpoint text may truncate, but the reconnect control stays
  icon-only and aligned to the row center.
- Endpoint validation and reconnect failures MUST use the existing `Last Error`
  row in the daemon/session group. Do not add a separate `endpointError` row,
  paragraph, alert block, or extra reserved-height error element under the
  endpoint input; it duplicates `Last Error` and destabilizes the compact row
  layout.
- Server version and protocol version are one compact metadata row, for example
  `Server 0.1.0 / Protocol 1.0`. They MUST NOT consume two separate rows in the
  narrow task pane unless localization makes the combined row unreadable.
- The tool summary is not just a count. It must expose the effective available
  tool list grouped by category, with each category independently expandable and
  collapsible. Category collapsed state is purely UI state and does not change
  tool availability.
- Available tools and tool permissions SHOULD be represented by one grouped
  `Tools` surface instead of two repeated blocks. The default collapsed view
  shows category names and compact enabled counts in `X/Y` form. Expanding a
  category shows tool names, descriptions, side-effect badges, and per-tool
  enable/disable controls. A separate read-only available-tools block is allowed
  only if it does not repeat the same detail already visible in the permission
  controls.
- The grouped `Tools` surface has a top-level capability mode control with three
  states: `Read`, `Write`, and `All`. `Read` enables only read-only tools;
  `Write` enables read tools plus mutating non-destructive tools; `All` enables
  every supported tool, including delete/destructive tools. This mode is a bulk
  permission control over the current session's tool set; individual tool
  toggles may further disable tools inside the selected ceiling.
- Each category row orders controls as: category label, right-aligned enabled
  count (`X/Y`), then the category toggle. The enabled count must sit directly
  before the toggle and align with counts in other categories. Clicks on the
  category toggle or per-tool toggles must not collapse/expand the category.
- Current task appears above history because it explains why Word may be busy.
- History entries are compact rows with expandable details.
- Settings and endpoint override live inside the top block as inline editing
  rows anchored to the metadata they edit. Opening settings MUST NOT append a
  new detached block at the bottom of the task pane, and SHOULD NOT create a
  separate framed settings card when edit-in-place rows can fit.

### 6.2 Required sections

The task pane MUST show:

1. **Daemon connection**
   - Daemon endpoint.
   - Connection state: `connected`, `connecting`, `reconnecting`, `disconnected`,
     or `failed`.
   - Server version after registration and protocol version in one row.
   - Last connection error and retry countdown when disconnected.

2. **Current document session**
   - Session ID.
   - Document title and filename.
   - Host app/version/platform.
   - Protection summary and dirty/read-only state when available. Normal editable
     documents MUST NOT be shown as `unknown`. If a host API confirms read/write
     access, show `Editable`; if document or worksheet protection is enabled,
     show the specific protected/readonly state. If a host does not expose a
     reliable dirty/saved signal, omit dirty state from the primary summary or
     move `not reported by host` to diagnostics instead of showing
     `Dirty: unknown`.
   - Available tool count in compact `X/Y` form when shown next to grouped
     controls.
   - Available tools grouped by capability category, with each category
     independently expandable/collapsible and names visible on expand or
     inspect. A bare label such as `27 Tools` is insufficient because it hides
     what the current document can actually do.

3. **Current task**
   - The command currently dispatched to this add-in runtime, if any.
   - Tool name.
   - High-level user intent when supplied.
   - Start time, elapsed time, and deadline.
   - Cancellation state when a cancel request has been received.

4. **Task history**
   - The most recent 20 completed commands for this add-in runtime.
   - High-level summary: tool name, user intent when supplied, duration, result.
   - Result: `success`, `failure`, `cancelled`, or `timeout`.
   - On failure: `office_mcp_code`, short error message, retriable flag, and
     `partial_effect` when known.

### 6.3 Settings

The task pane MUST keep the existing endpoint override capability. It SHOULD be
presented as inline edit-in-place rows inside the top connection/document block
rather than as the primary screen, a detached bottom block, or a large separate
settings card. Saving an endpoint override triggers reconnect and clearly
reports validation errors.

The endpoint field uses `type="url"`, `inputmode="url"`, a visible label with a
clickable hit target, a meaningful `name`, `autocomplete="off"`, and
`spellcheck="false"`. Its placeholder shows an example endpoint and ends with
an ellipsis. The save action stays enabled until the request starts, then shows
a saving state. Validation errors are inline, explain the required scheme and
host, and focus the field. Unsaved endpoint edits warn before closing the
settings view or navigating away.

The settings panel MUST include per-tool permission controls for the current
document session:

- Every advertised tool is listed by full tool name, category, side effect, and
  a short human-readable description.
- Tool permission controls are grouped by the same capability categories as the
  available-tools summary. Each category can be expanded/collapsed separately
  and shows a right-aligned compact `X/Y` count while collapsed; the phrase
  `Enabled X of Y` is too wide for the task pane and must not be used in category
  rows.
- The `Tools` header may show an aggregate `X/Y` count, but category rows show
  only `X/Y`. Counts align to the right edge of the row and appear immediately
  before the category toggle.
- The top-level `Read` / `Write` / `All` capability toggle controls the maximum
  permission tier for the whole tools surface. Side-effect classification comes
  from tool metadata: `read` for `Read`, `edit` or `comment` for `Write`, and
  `destructive` for `All`. Selecting a lower tier disables incompatible tools
  and removes them from the effective `available_tools` until the user selects a
  broader tier again.
- The available-tools summary and permission controls SHOULD be merged into one
  grouped tools surface to avoid repeated lists. In normal mode, controls may be
  read-only-looking but still reveal tool names; in settings/edit mode, the same
  rows expose toggles without duplicating the block.
- Each tool has an independent enable/disable toggle. Read-only tools and
  mutating tools are both controllable; mutating tools should be visually
  distinguished.
- Category-level toggles may be provided for convenience, but they are shortcuts
  only. The saved state is per tool.
- Disabled tools are removed from the session's effective `available_tools` and
  must not be invoked by the daemon for that session. If a call races with a
  local permission change, the add-in returns `TOOL_DISABLED_BY_USER`.
- Permission changes are persisted per host app and document session runtime in
  add-in storage, and reconnecting the same runtime preserves them. The control
  does not claim to change Office document permissions or IRM rights.
- The UI must show enough context for a user to understand the impact of a
  disabled tool, without exposing document body content or full tool arguments.

### 6.4 Add-in responsive constraints

- Minimum supported width: 320 px.
- No horizontal scrolling at 320 px.
- Sections use content-based sizing with explicit max-height only where a list
  can grow. Empty current-task and recent-task states are compact rows, not
  large blank panels.
- Long document titles and client names use two-line clamp before truncation.
- Command IDs and session IDs use middle truncation with copy controls.
- Current-task elapsed time remains visible even when the task name wraps.
- Touch targets are at least 32 px high in the Office task pane.

## 7. UI state model

The daemon owns the canonical state for clients, sessions, in-flight commands,
and global recent history. The add-in owns document-local state and mirrors the
command currently dispatched to that runtime.

### 7.1 Command record

Every command that reaches the daemon router produces a bounded in-memory record:

```json
{
  "command_id": "55555555-5555-5555-5555-555555555555",
  "mcp_request_id": "msg-789",
  "client_id": "client-1",
  "client_name": "copilot-cli/1.0",
  "session_id": "44444444-4444-4444-4444-444444444444",
  "host_app": "word",
  "tool": "word.insert_paragraph",
  "user_intent": "add summary paragraph after intro",
  "status": "success",
  "started_at": "2026-06-14T12:00:00.000Z",
  "completed_at": "2026-06-14T12:00:00.147Z",
  "elapsed_ms": 147,
  "error": null
}
```

On failure, `error` contains the redacted error shape defined in
[06-error-model.md](06-error-model.md). `user_intent` is optional and may be
absent for clients that do not provide it.

### 7.2 Retention limits

- Daemon global recent history: 10 completed commands.
- Daemon per-document expanded history: 10 completed commands per session.
- Add-in task pane history: 20 completed commands for that runtime.
- Current tasks are removed from the in-flight list immediately after they enter
  completed history.

### 7.3 Redaction

The UI state API MUST redact:

- Document body text.
- Inserted or replacement text.
- Base64 image data.
- File contents.
- TLS certificate passphrases and any future sensitive local configuration
  values. These are daemon transport/configuration values, not add-in-channel
  protocol fields.
- Any sensitive fields accidentally introduced by dependencies or platform
  adapters; the UI contract must not depend on such fields for access.
- Raw stack traces by default.

The daemon MAY expose a copyable diagnostic bundle later, but that is not part
of v1 UI.

## 8. Component requirements

### 8.1 Status badges

Status badges use text, icon, and color. Required states:

| State | Badge text | Tone |
|---|---|---|
| Daemon up | `Up` | success |
| Daemon degraded | `Degraded` | warning |
| Daemon down | `Down` | danger |
| Connected | `Connected` | success |
| Connecting | `Connecting…` | neutral |
| Reconnecting | `Reconnecting in <n>s` | warning |
| Running | `Running` | accent |
| Succeeded | `Succeeded` | success |
| Failed | `Failed` | danger |
| Cancelled | `Cancelled` | neutral |
| Timed out | `Timed Out` | warning |

Loading and reconnecting text uses an ellipsis character in rendered UI.

### 8.2 Copy controls

Endpoints, IDs, config paths, and log paths should provide a copy button. The
button is icon-only only when it has an accessible name such as `Copy MCP
Endpoint`. Copy success is announced through a polite live region.

### 8.3 Error details

Error rows show:

- Human-readable summary.
- `office_mcp_code`.
- Retriable flag.
- Partial-effect value when known.
- Next step when one is known.

Raw stack traces are hidden by default. A future explicit diagnostics mode may
expose them after redaction.

### 8.4 Confirmation dialog

Quit is destructive because it disconnects clients and add-ins. The dialog MUST
show current client count, document count, and running task count. The primary
action label is `Quit Office MCP Control`; the secondary action is `Keep Running`.

## 9. Implementation review checklist

Before release, review UI code for:

- Semantic buttons, links, labels, and tables.
- Visible focus states with no unreplaced `outline: none`.
- `aria-label` on icon-only controls.
- `aria-live="polite"` for async status, copy, validation, and command updates.
- Reduced-motion support and explicit transition properties.
- `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat`, and `Intl.NumberFormat` for
  displayed times and numbers.
- Long-content handling for document titles, paths, endpoints, IDs, client names,
  user intent, and error messages.
- Virtualization or `content-visibility` for lists above 50 rows.
- Correct light, dark, and high-contrast themes.
- No document body text, inserted text, image data, sensitive local
  configuration values, or raw stack traces in UI snapshots.
